# 架构卫生规则

## 上下文饱和检查

开始任何非琐碎代码变更前，确认待修改区域是否命中以下阈值：

- 单个手写代码文件超过 800 行；Markdown 文档不适用该阈值。
- 单类超过 30 个公开方法。
- 单函数超过 80 行，或圈复杂度超过 15。
- 路由文件超过 20 个路由。
- 即将复制超过 40 行已有代码。
- 同一职责在 3 个及以上文件中通过字符串字面量耦合。
- 同一实体存在多套手写 schema、DTO、ORM 或序列化表示且缺少单一真相来源。
- 生命周期状态字段的合法转移散落在多个分支中，或集中状态机只登记状态集合但缺完整转移表。
- 跨表、跨文件、跨服务或 DB + 文件系统更新缺少幂等键、回滚策略，或在事务块内执行不可回滚副作用。
- 旧 facade、兼容 shim、历史路径、旧数据格式、过期 API、不可达 UI 分支或调试脚本仍参与主流程。

任一阈值命中时，计划中说明拆分、抽象、注册表、状态机或 schema 单一来源的处理方式；不得以“匹配现有风格”或“检查未报警”为理由继续追加同类债务。

## 用户可见功能

设计文档、需求说明或治理方案中的能力词不能直接映射为 UI 按钮、状态机、后台任务、API 路由或 schema 字段。新增用户入口前必须说明：

- 用户要完成的任务、最短路径和必须主动决策的节点。
- 是否可用已有确认、执行、编辑、重新生成、撤销或回滚动作表达同一意图。
- diff、快照、日志、事件、版本和结果详情是否足以承载审计诉求。
- 每个新增按钮、状态、任务、API 或 schema 字段对应的明确用户动作。

## Agent 输出字段所有权

涉及 Agent、结构化 formatter、LLM 输出格式化或提示词输出要求时，先做字段所有权矩阵：

- backend-owned：任务输入、DB row、route 参数、store context、scheduler context 或确定性 projection 已知的字段，不要求 LLM 输出。
- agent-owned：只有 Agent 结果才能提供的业务语义，例如分析结论、建议动作、证据摘要和风险说明。
- boundary-owned：DB、HTTP、文件、日志和观测等序列化边界字段。

Prompt 和结构化输出契约只能要求 agent-owned 输出字段。Backend-owned 字段可以作为输入上下文帮助 grounding，但不能成为 Agent 输出结构的一部分。
