# 验证规则

## 功能验证

- Python 运行优先使用仓库本地 `.venv/bin/python`；项目未配置 `.venv` 时说明替代解释器和风险。
- 后端变更默认运行相关 `pytest`；跨模块变更运行项目声明的完整测试或等价命令。
- 前端变更默认运行项目声明的构建命令，必要时补充浏览器验证。
- 用户可见页面状态变更必须验证空态、成功态和失败态错误详情。
- Docker 持久化相关变更必须检查项目覆盖层声明的默认运行态路径，以及历史卷路径的迁移或兼容策略。
- Runtime/env 相关变更必须验证 mode 选择、env 示例边界、私有 env ignored 状态、模型凭据不进示例文件、启动日志字段和 README/docs 术语一致性。

## 架构验证

完成后必须说明：

- 是否新增超过阈值的代码文件、函数、类或路由模块。
- 是否新增超过 40 行的重复代码块。
- 是否新增生命周期状态字面量分散、缺转移表的状态机、schema 双轨、跨事务中间态或硬编码分类字符串。
- 如触发替换旧设计模式，旧 facade、兼容 shim、历史路径、重复实现、不可达分支或过期 API 是否已按计划删除、迁移或明确保留。
- 如公开 API、配置/env、持久化数据或旧入口被改变，是否补充迁移说明、文档更新和针对新契约的测试。
- 是否运行项目覆盖层声明的治理硬门；项目未配置时说明替代验证与缺口。

## 测试纵深

- 生命周期状态字段变更：至少补充 1 个非法状态转移或非法输入测试。
- 并发资源变更：至少补充 1 个重复执行、竞争或部分失败测试。
- 外部输入变更：至少补充 1 个异常、恶意或越权输入测试。
- AI / 自动化输出契约变更：至少补充 1 个 hostile backend-owned 字段污染测试。
- Happy path 只能证明主流程可用，不能替代边界和失败场景验证。
