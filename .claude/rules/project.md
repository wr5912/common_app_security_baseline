# 项目统一规则

## 执行顺序

- Analyze 阶段先阅读相关代码、配置、测试、文档和报错，能从仓库确认的事实先用只读方式确认。
- Plan 阶段说明涉及文件、逻辑修改点、验证方式、Docker 卷路径、架构阈值命中情况和治理硬门。
- Execute 阶段只做已确认目标需要的修改；发现无关问题时记录并说明，不顺手扩大范围。
- Verify 阶段对照原始目标验证结果，运行相关测试、构建、类型检查或人工验收。

## 四项工程行为

- 三思而后行：显式说明事实、假设、歧义和权衡。
- 简洁优先：用最少实现解决当前目标，不添加未要求的抽象、依赖或扩展点。
- 精准改动：每一行修改都应能追溯到用户目标，普通小改不重排无关格式。
- 目标驱动：把任务转成可验证成功标准，完成后说明验证结果和残余风险。

## 质量优先

当目标涉及重构、去除旧设计、优化架构、提高代码质量，或发现旧 facade、兼容 shim、历史路径、重复实现、schema 双轨、状态分散、过期 API、不可达分支等旧债信号时，优先级为：代码质量 > 新设计/框架/架构 > 旧模式兼容或保留。

触发该模式时，计划中列出删除、迁移、保留清单，并说明公开 API、配置/env、持久化数据、文档、测试和内部兼容层如何处理。

## Claude Code 面向

- 常驻事实放在 `CLAUDE.md` 或 `.claude/rules/`；多步骤流程放进 skill；独立大范围搜索或审查放给子代理。
- 不把个人偏好、账号、密钥、本机路径、私有 MCP 或运行态数据写入共享配置。
- Claude Code 只读取 `CLAUDE.md`，不读取 `AGENTS.md`；需要复用其他 agent 说明时在 `CLAUDE.md` 中显式 `@` 导入。
