# 优化效果评估

评估目标是证明“优化后更好”，不是证明“配置更多”。

## 最小 eval 集

至少准备 3 类真实场景：

1. 配置优化请求：应先审计和删重，不默认新增。
2. Agent / AI 输出契约失败：应先做证据链、字段所有权和执行动作分类。
3. Skill 触发问题：应判断 description、触发词和 skill 边界，而不是修改无关配置。

## 评分维度

| 维度 | 通过标准 |
| --- | --- |
| 触发准确 | 相关任务会使用 skill，不相关任务不误用 |
| 配置经济性 | 建议中 `delete/merge/move-*` 优先于新增 |
| 归因质量 | 先分类 failure，再给动作 |
| 表面选择 | 配置面和职责匹配 |
| 验证闭环 | 每条建议有可运行检查或明确人工验收 |
| 安全边界 | 不写入私有路径、凭据或项目私货到通用层 |

## Judge 使用门槛

优先使用静态脚本、grep、pytest 或真实任务对比。只有以下条件满足时才设计 LLM judge：

- failure mode 无法用确定性检查覆盖。
- 已有足够人工标注样本。
- 单个 judge 只判断一个明确失败模式。
- judge 需要用 held-out 样本校准，并记录误报/漏报。
