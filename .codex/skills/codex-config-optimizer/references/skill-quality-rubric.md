# Skill 质量准则

用于审计 `SKILL.md`，重点不是写得多，而是触发准、上下文轻、验证清楚。

## P0 检查

- `name` 使用小写 kebab-case，含义具体。
- `description` 说明“什么时候使用”，包含用户自然会说的触发词。
- `SKILL.md` 聚焦一个任务，不混入多个无关流程。
- 详细规则进入一级 references；不要深层嵌套引用。
- 脚本用于确定性、重复性强或容易写错的操作。
- 输出有明确输入、动作、验证和失败处理。

## 常见问题

| 问题 | 症状 | 建议 |
| --- | --- | --- |
| description 像功能介绍 | skill 不触发或触发错 | 改成触发条件和关键词 |
| 重复通用知识 | 上下文变重，行为无提升 | 删除，保留项目特有坑点 |
| 过度约束路径 | Agent 无法适应真实场景 | 约束目标和检查点，少写死顺序 |
| 全塞进 `SKILL.md` | 触发后上下文膨胀 | 拆到 references 或 scripts |
| 缺 eval | 不知道是否变好 | 至少准备 3 个真实失败场景 |

## 审计输出

每个问题按 `P0/P1/P2` 标注：

- `P0`：会导致 skill 不触发、误触发、泄露私有信息或执行危险动作。
- `P1`：会导致上下文膨胀、重复、输出不稳定或验证缺失。
- `P2`：可读性、组织和泛化质量问题。
