# 服务持久化机制对比

> 跨平台基线方法论页（`os: cross`）：把“服务常驻持久化”在 Windows 与 Linux 上的对应位置放在一起，便于误报核验与横向迁移判断。

## 1. 基线说明

“以服务/守护方式开机自启常驻”是正常软件的普遍行为，但也是攻击者最常用的持久化手段。判断要点在两边是一致的：**谁写的、写到哪、ExecStart/ImagePath 指向什么、启用后是否外联或落地文件**。

## 2. 机制对应

| 维度 | Windows | Linux |
|---|---|---|
| 启动方式 | [[Windows Service]] / [[Scheduled Task]] | [[systemd Service]] / [[Cron Job]] |
| 持久化位置 | [[HKLM_SYSTEM_CurrentControlSet_Services]] | [[systemd unit 持久化]] / [[Crontab 持久化]] |
| 启动命令字段 | `ImagePath` | `ExecStart` |
| 常见父进程 | `services.exe` | `systemd`(PID 1) / `cron` |
| 运行账户 | `ObjectName`（LocalSystem 等） | `User=`（多为 root） |

## 3. 共性高风险信号

```text
启动命令指向用户目录 / 临时目录（/tmp、/dev/shm、%TEMP%）
启动命令包含脚本解释器或下载执行（sh/bash/powershell/curl|bash）
写入位置异常或为用户可写目录
新建后立即外联或落地文件
名称伪装系统服务 / 系统 unit
最近修改时间与变更窗口不符
```

## 4. 误报条件

```text
软件包安装 / 升级写入（apt、yum、MSI、安装器）
运维部署、容器编排注册
合法软件的更新器与心跳外联
```

## 5. 推荐处置

对比历史配置与基线，关联写入进程、账户、变更窗口与后续行为；两边都优先表达上下文和证据需求，不仅凭“新增了一个服务/unit”定恶意。
